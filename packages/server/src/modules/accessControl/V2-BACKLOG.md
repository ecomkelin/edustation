# v2 Backlog (V2-BACKLOG.md)

> PoC 阶段**不做**的需求汇总。新需求来时先看这里：是否已经规划过？
> 进入 v2 开发时，按优先级从 P0 开始逐个实现。

---

## 优先级总览

| 优先级 | 含义 | 何时做 |
|--------|------|--------|
| **P0** | 业务阻塞 / 强合规 | 1-3 个月内 |
| **P1** | 明显业务价值 | 3-6 个月 |
| **P2** | 体验优化 | 6-12 个月 |
| **P3** | nice-to-have | 12 个月+ |

---

## P0：业务阻塞 / 强合规

### P0-1：snapshot 30 天自动清理 cron

- **现状**：PoC 走手动 SQL 清理
- **目标**：每日凌晨 3 点自动清
- **实现**：`packages/server/src/jobs/cleanupAccessSnapshots.js`（新增）+ `startupMigrations.js` 注册
- **工作量**：0.5 天
- **触发条件**：上线后 1 个月内必须有

### P0-2：FaceProfile polymorphic 拆 3 表

- **现状**：单表 polymorphic + `refPath: 'subjectType'`，部分查询性能差
- **目标**：拆 `StudentFaceProfile` / `UserFaceProfile` / `PickupFaceProfile`
- **实现**：
  - 3 个新 model
  - `FaceProfile` 标记 deprecated
  - 迁移脚本：按 subjectType 分发
  - service 层先兼容，半年后切流量
- **工作量**：3-5 天
- **触发条件**：流水量 > 100k/天后优化

### P0-3：撤回时同步清除设备本地白名单

- **现状**：`driver.removeFaceProfile` noop，撤回后设备仍能识别
- **目标**：通过厂商 SDK API 远程清除
- **依赖**：厂商提供 SDK（PoC 阶段未采购 SDK 版）
- **工作量**：2-3 天 / 厂商
- **触发条件**：≥ 2 家机构使用 + 用户投诉"我撤回了还能开门"

### P0-4：与 LessonAttendance 联动（时间窗匹配）

- **现状**：门禁只记进出，**不**写考勤
- **目标**：识别到校 + 时间窗内有 LessonSchedule → 写 LessonAttendance.status='present'（或新增 status='face_recognized_arrived'）
- **实现**：
  - `AccessEvent` 新增 `linkedLessonAttendance` 字段
  - service 端在 recordEvent 后查 LessonSchedule（subject + start±15min）匹配
  - LessonAttendance 新增 status='present_via_face' 区分手动打卡
- **工作量**：3-5 天
- **触发条件**：用户主动要求"门禁自动消课"

## P1：明显业务价值

### P1-1：unipush 推送（家长端"孩子到校"通知）

- **现状**：未实现
- **目标**：学员到校 → 家长微信/APP 推送"孩子已到校"
- **依赖**：unipush 厂商认证 + AppID 申请
- **实现**：`modules/accessControl/hooks/unipushOnArrival.js`（新增）
- **工作量**：1-2 天
- **触发条件**：机构签约 unipush 后

### P1-2：考勤陌生人告警（教务短信）

- **现状**：仅入库，无通知
- **目标**：陌生人连续出现 3 次 / 1 小时内 → 教务短信告警
- **实现**：`modules/accessControl/jobs/strangerAlert.js`（每 5 分钟跑）
- **工作量**：1-2 天
- **依赖**：短信厂商 API
- **触发条件**：机构签订短信服务

### P1-3：设备本地白名单自动同步（解决"断网识别"）

- **现状**：`driver.syncFaceProfile` noop，依赖一体机自身同步策略
- **目标**：FaceProfile 录入/撤销时主动推设备
- **实现**：
  - 重试队列（`jobs/syncFaceProfileQueue.js`）
  - 退避策略（指数退避，最多 5 次）
  - 失败告警
- **工作量**：3-4 天
- **依赖**：厂商 SDK
- **触发条件**：机构网络不稳 + 投诉"断网时识别失败"

### P1-4：多机构支持（平台超管视角）

- **现状**：所有 model 按 `org` 隔离，但平台超管无跨机构 dashboard
- **目标**：平台 dashboard 看到所有机构的设备数 / 流水量 / 陌生人占比
- **实现**：`/api/v1/reports/access-control-platform`（聚合管道，跳过 `requireOrg`）
- **工作量**：2-3 天
- **触发条件**：机构数 ≥ 3

### P1-5：hikvision / dahua driver 真实实现

- **现状**：stub
- **目标**：覆盖海康 / 大华一体机
- **依赖**：采购对应设备 / 厂商协议文档
- **工作量**：3-5 天 / 厂商

## P2：体验优化

### P2-1：前端看板"陌生人占比"折线图

- 现状：Admin 端只有原始流水列表
- 目标：按日聚合陌生人占比，ECharts 折线
- 工作量：1-2 天

### P2-2：批量上传人脸（CSV 名单 + 压缩包）

- 现状：单条 enroll
- 目标：上传 CSV (student_id, name) + zip (face1.jpg, face2.jpg...) → 批量入库
- 工作量：2-3 天
- 触发条件：开学季录入大量新学员

### P2-3：接送授权二维码（现场扫码授权）

- 现状：客户端表单
- 目标：生成一次性二维码（10 分钟有效）→ 现场扫码授权
- 工作量：1-2 天
- 触发条件：家长投诉"我没法远程帮父母授权"

### P2-4：实时通知面板（家长端"刚刚识别"）

- 现状：只查历史
- 目标：active student 刚到校 → 客户端弹"您的孩子已到校"
- 工作量：1 天
- 依赖：unipush

### P2-5：设备健康看板（心跳 / 固件 / 电池）

- 现状：仅 Admin 设备列表显示 lastHeartbeatAt
- 目标：单独 dashboard，按"5min 无心跳"标红
- 工作量：1-2 天

### P2-6：温度检测 / 戴口罩识别（依赖设备能力）

- 不强制做；按机构需求 + 设备能力定
- 工作量：1 天（如果设备支持）

- 现状：门锁
- 目标：闸机 + 双向通行
- 工作量：硬件 + 软件 5+ 天

### P3-9：访客模式（一次性二维码）

- 现状：未实现

- 工作量：2-3 天

## 已知技术债

| 债 | 影响 | 何时还 |
|----|------|--------|
| polymorphic FaceProfile 性能 / 索引脏 | 百万级流水后查询慢 | P0-2 |
| driver 抽象不完整（远程开关门未实现）| 只能走设备本地策略 | P1-3 |
| webhookAuth 5min 窗口 | 设备时钟漂移敏感 | 改 10min |
| AccessEvent 与 LessonAttendance 完全解耦 | 业务上希望联动 | P0-4 |
| 撤回级联不走事务 | 并发撤回时可能漏 | P0-3 一起做 |
| 自动清理 snapshot cron 缺失 | snapshot 越积越多 | P0-1 |
| 第三方接送人脸录入 UI 缺失 | PoC 走纸质 | 等机构反馈 |
| 多机构 dashboard 缺失 | 平台超管不便 | P1-4 |

## 已拒绝的需求

> 这些是"看似合理但实际不该做"的需求，列在这里防止下次又被提起。

| 需求 | 拒绝原因 |
|------|---------|
| 用人脸识别做员工考勤 | 已有 SaaS 标准方案，硬件绑定会复杂化离职/转岗 |
| 学员人脸 + 接送家长人脸 合成"家庭档案" | 数据治理复杂度高；现 model 已够用 |
| 把 AccessEvent 同步到学校公众号 | 涉及第三方对接 + 同意书条款变更 |
| 接送授权自动续期 | 业务风险高（家长可能不希望自动续） |
| 把人脸识别门禁数据共享给"安全联盟" | 违反 PIPL 23 条，需法定职责评估 |
| 用 AccessEvent 数据训练 AI 模型 | 需重新签同意书 + 单独审批 |

## 何时启动 v2

判断标准：

- [ ] PoC 至少 2 家机构稳定运行 3 个月
- [ ] 用户提了 ≥ 3 个 v2 需求
- [ ] 流水量 > 100k/天
- [ ] 团队人力允许（v2 工作量 20+ 天）

满足任意 2 项即可启动 v2 sprint。
